# demo2opt 与中控 TPT/TPT2 差距评估（工业试点可用）

更新时间：2026-03-07

## 1. 评估基线（2026-03-07）

### 1.1 当前系统定位
- 当前系统是“规则预筛 + 人机确认 + LLM诊断 + 报告输出”的工业智能分析系统，已具备演示级闭环，但尚未达到“可规模复制的工业试点平台”。
- 关键代码形态：
  - 在线分析主流程：`src/api/ws/server.py`
  - 语义分析与规则：`src/services/data_semantics.py`
  - 决策建议：`src/services/decision_service.py`
  - 报告生成：`src/utils/report_generator.py`

### 1.2 测试基线（本次修复后）
- `pytest -q`：42 passed, 1 skipped（`tests/test_progress_api.py` 默认按集成测试跳过）。
- `npm run test:e2e -- --project=chromium --reporter=line`：13 passed。

### 1.3 “工业试点可用”定义（本评估采用）
- 单装置可持续运行：至少 4-12 周稳定运行，无高频人工救火。
- 可回溯：关键决策、输入数据、模型输出可审计。
- 可回退：对控制策略有明确降级与人工接管机制。
- 可运维：失败可定位、可恢复，具备基本 SLA 与值班支持流程。

## 3. 与 TPT/TPT2 差距清单（P0/P1/P2）

### P0（必须先补齐，影响试点成败）
- 现状：
  - 以规则/语义判断为主，缺少独立“可验证”的时序预测服务层（统一训练、回测、漂移监测未成体系）。
  - 任务持久化以本地 JSON 为主，缺少数据库级一致性、并发控制与恢复策略。
  - 安全治理薄弱（本轮未扩展安全改造，仅修测试阻断）。
- 目标：
  - 建立“预测服务 + 回测评估 + 漂移告警 + 策略回退”最小闭环。
  - 用数据库和可追踪任务状态替代文件级任务管理。
  - 实现最小鉴权、密钥托管、审计日志链路。
- 建议动作：
  - 0-4 周优先完成预测服务 MVP（含离线评估）和任务存储改造。
  - 将策略执行固定为“建议优先 + 人工确认”，保留强制回退。

### P1（试点扩展阶段）
- 现状：
  - 当前流程偏“分析与报告导向”，缺少面向控制系统的标准化接口层。
  - 缺少跨装置迁移评估与配置模板体系。
- 目标：
  - 建立控制接口抽象层，支持多厂站接入。
  - 建立“装置模板 + 指标模板 + 策略模板”复制机制。
- 建议动作：
  - 5-12 周推进多装置 PoC，对比不同工况下误报、提前量、收益稳定性。

### P2（平台化阶段）
- 现状：
  - 缺少完整 MLOps（数据版本、模型注册、灰度发布、在线监控）。
  - 缺少标准化商业化运维体系（SLA、升级策略、故障演练）。
- 目标：
  - 形成可规模复制的工业 AI 平台能力。
- 建议动作：
  - 3-6 月建立模型生命周期管理、灰度发布和多租户运维规范。

## 4. 分阶段路线图（可交付）

### 0-4 周（试点可启动）
- 交付：
  - 时序预测 MVP（至少 2 类模型 + 统一评估报告）。
  - 任务状态数据库化与失败恢复机制。
  - 试点运行手册（回退策略、人工接管、异常处置）。
- 验收：
  - 连续 2 周稳定运行，关键任务链路无阻断。

### 5-12 周（试点可扩展）
- 交付：
  - 多装置模板化接入能力。
  - 跨装置评估报告（准确率、提前量、误报率、节能收益）。
  - 最小权限和审计落地。
- 验收：
  - 至少 2 个装置可复制部署。

### 3-6 月（迈向准产品化）
- 交付：
  - 模型注册/发布/回滚规范。
  - 在线监控与告警体系。
  - 服务级稳定性与运维指标（SLA）报告。
- 验收：
  - 具备平台化复制能力与可持续运维能力。

## 6. 结论
- 当前 demo2opt 已恢复“工程测试可用”状态，但与 TPT/TPT2 的核心差距仍在“模型平台化、控制闭环、安全合规、规模化运维”四个方面。
- 对“工业试点可用”目标，建议按本文件路线优先推进 P0；若 P0 未闭合，后续扩展将高概率遇到稳定性与安全瓶颈。
