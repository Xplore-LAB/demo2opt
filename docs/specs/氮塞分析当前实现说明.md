# 氮塞分析 Demo 当前实现说明

## 1. 时序数据输入与存储
- 当前 Demo 不直接让前端解析原始 Excel，而是由后端先整理进 SQLite。
- 前端先调用 `/api/nitrogen-demo/meta` 获取总行数、时间范围、字段元数据。
- 再调用 `/api/nitrogen-demo/data` 按 `start_ms / end_ms / max_points` 拉取当前时间窗口的数据。
- 这样前端只拿当前展示需要的一段数据，不会一次加载全量时序点。
- 精选病例是单独链路，优先直接读取 `raw_5s_file`；只有细粒度文件不可用时才回退 `resampled_1min_file`。

## 2. 为什么用 SQLite
- 目标是单机演示、轻量部署、少依赖，所以优先选 SQLite。
- 这个场景的主要读法是按时间范围查询，SQLite 已够用。
- 不需要额外起数据库服务，便于本地调试和打包分发。
- 对 Demo 来说，SQLite 比前端直接读大文件更稳，也比每次重新解析原始文件更快。

## 3. 时序数据链路现在怎么走
- 数据表核心是 `samples`，附带 `column_meta` 做字段映射。
- 后端按别名把测点映射成 `AI705 / AI701 / FI702 / FIQC701 / FIC101 / BALANCE` 等前端统一字段。
- 返回给前端的数据已经是结构化记录：时间戳、显示时间、各测点数值。
- 前端再基于这些记录做趋势图、窗口扫描、疑似区间合并和故障树分析。

## 4. 为什么前端大数据量时容易卡顿
- 主要原因不是“图表组件本身”，而是时序点过多时，图表重绘和联动更新成本很高。
- 页面里同时存在趋势图、区间高亮、画刷选区、故障树、状态卡片，任何一步更新都会触发重新渲染。
- 如果直接把全量时序点都塞给前端，切换测点和缩放窗口时会明显变慢。

## 5. 现在怎么减轻前端卡顿
- 后端接口限制 `max_points`，先做一次按时间范围裁剪。
- 前端还有 `sampleRows()` 做二次抽样。
- `buildChartRows()` 会根据图宽和重点测点保留起止点、极值点，减少无效点。
- 疑似区间扫描放到 `Web Worker` 中执行，避免阻塞主线程。
- 精选病例细窗口会放宽前端抽样限制，优先保证 `AI705` 等关键曲线的局部细节。
- 趋势图最大回看时间尺度限制为 7 天，当前关闭滚轮自由缩放和框选放大，改用固定时间尺度按钮与时间进度条定位，避免连续刷新导致卡顿。
- 趋势图纵坐标按工艺默认观察尺度固定，横向拖动或切换时间尺度时不再随当前窗口自动缩放，避免把纵轴变化误判成过程突变。比如 `AI705` 默认按 `95-100%` 展示，只有实际数据越出该范围时才向外扩展；归一化趋势也按同一套固定范围归一化，不再按当前可见窗口重新拉伸。前端单独维护 `fixedMetricBounds`，不再用当前显示窗口 `archiveRows` 反算 Y 轴范围。

## 6. 程序主流程
- 页面进入后，前端先取 `/api/nitrogen-demo/meta` 和 `/api/nitrogen-demo/data`，拿到当前时间范围内的时序数据。
- 如果用户切换到精选病例，页面会直接加载病例配置中的指定时间段窗口数据，不再依赖历史库按大窗口回查。
- 页面支持时间进度条拖动定位，常用尺度按钮会切换当前窗口长度并复用同一套时间游标状态。
- 趋势图提供 `1h / 6h / 12h / 24h / 48h / 72h / 7d` 常用时间尺度按钮；为了保证交互流畅，图面本身不再响应滚轮自由缩放和框选放大。
- 浏览趋势和识别判断已解耦：拖动时间、点击常用尺度或播放时间轴只刷新曲线，并清空旧识别结果；只有点击“识别氮塞”才会扫描当前窗口、绘制红色疑似区间和标注。
- 时间进度条支持点击轨道直接跳转，并在悬浮时提示当前对应时间。
- 前端把时序数据按固定窗口长度切成很多滑动窗口，每个窗口单独判断一次。
- 每个窗口会计算关键测点的 `mean / min / max / delta`，再生成一个窗口级结果。
- 所有窗口结果汇总后，会把 `suspected=true` 的窗口合并成连续区间，显示为“疑似氮塞区间”。
- 识别完成后默认选中最明显的疑似区间，依据是风险等级、AI705 下凹深度和视觉置信度，而不是简单选最后一个时间窗口。
- 如果用户点击“开始分析”，前端会把当前选中的窗口信息发给 `/api/nitrogen-agent/analyze`。
- 后端先返回阶段结果驱动页面，再在完整分析阶段调用大模型生成结构化结论。
- 趋势图按固定时间尺度切换窗口，避免滚轮连续缩放造成频繁数据刷新和图表重绘。

## 7. 当前怎么判定“疑似氮塞”
- 当前识别链路固定为：数据窗口 → 基线估计 → 下凹形态识别 → 相对下降幅度 → 持续时间 → 等级划分 → 自动标注 → 原因分析。
- “是否疑似氮塞”和“疑似等级”只由 `AI705` 当前工作点附近的下凹形态决定，不再把联动测点数量作为触发条件。
- 识别阶段看 `AI705`：
  - 连续触发：窗口内连续至少 3 个点低于初始基线，且最低点相对基线下滑足够深。
  - 尖峰触发：单步下挫足够大，且整体下探足够深。
  - 视觉短时深谷：原始 AI705 出现明显单点或短时深谷、两侧恢复清楚时，先作为轻微疑似区间纳入结果。
  - 临近触发：还没到正式触发，但已经接近阈值。
- 等级划分依据为相对下凹幅度、持续时间、恢复比例、小谷数量和视觉置信度。
- `AI705` 在流程里只作为顶事件/形态证据，不再直接抬高某个原因分支的置信度。
- 辅助证据不参与疑似识别，只在用户点击“分析原因”后进入原因分析：
  - 先判断运行状态：稳态或动态/变负荷；动态过程按变负荷速率和时序匹配单独分析。
  - 稳态下按装置分块排查：`A 粗氩塔`、`B 主塔`、`C 空气系统`、`D 事件`。
  - `A 粗氩塔`：`FI702` 氩馏分流量偏高、`FIC701` 粗氩流量偏低。
  - `B 主塔`：氧气多抽、氮气少抽、`V3` 阀开过大、`AI701` 长时间偏高或短时间高高。
  - `C 空气系统`：空气少进、膨胀空气旁通量偏少。
  - `D 事件`：分子筛切换不平稳；动态过程另按变负荷分析。
  - 物料平衡用于判断目标负荷、目标氧气、目标液氮，以及空气/氧气/氮气相对基准值 `±1.25%` 范围内的偏低、正常、偏高和低/中/高偏离程度。
- 当前窗口最终会被标成 `suspected=true` 的条件是：`AI705` 正式触发或达到观察候选，不需要辅助联动证据补票。
- 风险等级目前按这套规则分：
  - 已疑似且证据多：`high`
  - 已疑似但证据一般：`medium`
  - 只有主触发或弱证据：`low`
  - 没达到疑似：`normal`

## 8. 物料平衡在程序里的位置
- 当前“物料平衡”已经单独作为步骤2存在。
- 目前程序会先计算 `BALANCE` 在窗口内的均值、变化量和波动范围。
- 如果绝对变化过大，或者窗口波动过大，就认为当前窗口存在物料平衡异常信号。
- 这一步当前只作为辅助校核，不直接决定是否为氮塞。

## 9. 当前怎么调用大模型
- 前端统一调用 `/api/nitrogen-agent/analyze`。
- 当前不是一次性只调一次，而是按步骤调用：
  - 步骤1：检查主触发
  - 步骤2：检查物料平衡
  - 步骤3：判断故障树分支
  - 步骤4：生成结论
- 前 3 步主要返回阶段状态和一句话结果，用来驱动页面展示，这几步默认不真正调用大模型。
- 第 4 步才进入完整分析，后端调用 `_call_agent_llm()` 请求大模型。
- 如果第 4 步成功，返回 `analysis_mode=llm_enhanced` 和 `llm_trace.status=succeeded`；如果失败，则回退到规则结果。

## 10. 大模型输入什么
- 大模型拿到的不是全量原始时序数据。
- 后端先把区间信息整理成结构化上下文，再发送给模型。
- 输入内容包括：
  - 区间时间范围
  - 风险等级
  - 触发测点
  - 基础判断依据
  - 原因分支
  - 物料平衡复核说明
  - 缺失信息
  - 复核测点
  - 故障树步骤

## 11. 大模型输出什么
- 大模型必须返回结构化 JSON，不是自由文本长回答。
- 重点输出字段包括：
  - `conclusion`
  - `summary`
  - `top_event_judgement`
  - `basis`
  - `evidence_nodes`
  - `fault_tree_path`
  - `branch_ranking`
  - `key_missing_data`
  - `cause_branches`
  - `material_balance_review`
  - `fault_tree_steps`
  - `action_advice`
  - `review_tags`
  - `missing_information`

## 12. 后端完整分析怎么走
- `/api/nitrogen-agent/analyze` 会先根据当前窗口生成规则结果，包括：
  - 顶事件判断
  - 证据节点
  - 故障树路径
  - 原因分支排序
  - 缺失数据
- 如果当前只是阶段调用，后端直接返回阶段标题、阶段说明、焦点节点和一句话结果。
- 如果进入完整分析，后端会把规则结果整理成结构化上下文，再请求大模型。
- 如果大模型成功，返回 `llm_enhanced`。
- 如果大模型失败，回退到 `rule_based`，页面继续展示规则结果，不中断流程。

## 13. 现在页面怎么展示分析过程
- 先显示疑似区间和当前状态。
- 开始分析后，按步骤推进故障树：
  - 当前步骤
  - 当前焦点节点
  - 相关分支卡片
  - 待补节点
  - 当前建议
- 这样不是最后一次性扔出完整结论，而是边推进边展示。

## 14. 为什么现在还会感觉“不够直观”
- 虽然已经有步骤推进，但节点之间的关系感还不够强。
- 当前节点、已完成节点、待判断节点之间的视觉差异还可以继续拉大。
- 图表趋势和故障树节点的联动还不够强，用户不容易一眼看出“哪个测点支撑哪个节点”。

## 15. 物料平衡这一步当前做到什么程度
- 目前已经单独放成“步骤2：检查物料平衡”。
- 现在先作为流程中的独立步骤存在，页面上会明确显示这是一个占位复核步骤。
- 具体方程、参与测点和定量判据还没有完全接入。
- 当前程序只是先看 `BALANCE` 指标的均值、变化量和波动范围，用来做辅助复核。
- 后端返回里会带 `material_balance_mode=placeholder_review` 和 `material_balance_formula_status=pending`，前端据此展示“正式方程待接入”。
- 依据是现有溯源逻辑文档里已明确：物料平衡适合作为辅助校核节点，但不能单独定位氮塞原因。

## 16. 当前大模型配置
- 当前氮塞分析后端会优先读取项目根目录的 `llm_configs.json`。
- 如果本地未配置，再回退读取 `.env` 中的 `LLM_BASE_URL / LLM_API_KEY / LLM_MODEL_NAME`。
- 当前本地已切换到：
  - 首选模型：`deepseek-v4-flash`
  - 首选 Base URL：`https://api.deepseek.com/v1/chat/completions`
  - 备用模型：`mimo-v2.5-pro`
  - 备用 Base URL：`https://token-plan-cn.xiaomimimo.com/v1`

## 17. 当前已知问题
- 如果大模型配置不可用，页面会回退到规则结果。
- 回退后容易看到“规则兜底结论”或本地摘要，用户会误以为系统没有真正分析。
- `MiMo` 网关当前已验证可连通，但在完整氮塞结构化分析提示词下，返回 JSON 仍有不稳定情况，所以本地优先级先切到 `DeepSeek`。
- 目前最需要继续优化的是：
  - 当前步骤更直观
  - 当前节点更明显
  - 图表与故障树联动更强
  - 大模型失败时提示更清楚

## 18. 后续建议
- 补全物料平衡方程，把步骤2从占位步骤变成真正可判定步骤。
- 增强故障树动态效果，让当前节点、上一节点、下一节点更清楚。
- 增加“节点 -> 对应测点”的映射展示，让用户直接看到依据。
- 优化大模型失败后的提示，不只显示“回退规则”，还要说明当前用了哪套兜底逻辑。
