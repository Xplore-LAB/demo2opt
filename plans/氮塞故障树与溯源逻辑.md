# 氮塞故障树与溯源逻辑

> 来源：2026-04-27 会议文字记录。  
> 整理原则：尊重原始会议记录，把“会议原文依据”和“执行扩展”分开写。  
> 说明：本文件不是最终技术方案，而是把会议中关于“氮塞检测后的溯源逻辑”单独整理成可推进任务。

---

## 1. 任务目标

把“检测到疑似氮塞以后应该怎么查原因”整理成故障树/诊断树，避免系统完全依赖大模型自由发挥。

会议中老师明确指出：当前检测相对简单，真正缺口是检测后的溯源逻辑。也就是说，不仅要判断“是否疑似氮塞”，还要回答：

- 问题可能在哪个系统或单元；
- 有哪些可能原因；
- 每个原因查哪些测点；
- 每个测点如何判断正常、偏高、偏低；
- 多个原因同时存在时如何排序；
- 输出建议时有没有数据和判据支撑。

---

## 2. 会议原文依据

### 2.1 检测后溯源逻辑不清楚

老师原文核心意思：

```text
检测现在相对来说比较简单，因为它是一个单变量的检测问题。
后面相当于是溯源，检测到了以后，首先溯一下这个问题到底在哪里。
搜索路径完不完整？有没有整理完整？
```

可确认结论：

| 内容 | 结论 |
|---|---|
| 检测本身 | 可以先做，难度相对低 |
| 溯源逻辑 | 当前没有整理完整 |
| 后续重点 | 梳理搜索路径和判断逻辑 |

### 2.2 每条建议不只是要数据，还要判断逻辑

你提出：具体建议需要数据支撑，例如 V3 阀开度偏大要给出 V3 阀数据。老师进一步追问：

```text
V3 阀开得算不算大，这个东西怎么判断？
每一条具体判据是不是已经量化？
过大怎么说过大？那就是量化的问题。
```

可确认结论：

```text
数据本身不等于判据。
建议必须绑定“数据 + 判断逻辑 + 判据”。
```

### 2.3 问答集还不是知识图谱

老师原文核心意思：

```text
你现在这个 question 还没有形成故障树分析。
问答集还没有到知识图谱这个过程。
知识图谱不是问答集。
问答集可以用于知识问答，也可以用于微调，但还没有转成结构化的诊断逻辑。
```

可确认结论：

| 对象 | 会议判断 |
|---|---|
| 问答集 | 已完成第一步，可用于问答/微调 |
| 故障树 | 当前需要补的一步 |
| 知识图谱 | 需要更系统化，不能直接等同问答集 |

### 2.4 故障树是推荐的组织方式

老师原文核心意思：

```text
氮塞可以作为一个顶事件。
第一层原因是哪几个原因？每个原因下面又是什么原因？
这就是故障树。
故障树里面还可以包括与或逻辑关系。
一般两层或者三层基本上就到底了。
```

可确认结论：

| 项目 | 会议结论 |
|---|---|
| 顶事件 | 可设为疑似氮塞 |
| 层级 | 2—3 层为宜 |
| 逻辑关系 | 可考虑与门/或门 |
| 末端节点 | 尽量落到测点、事件或判据 |

### 2.5 顺序不是最关键，覆盖分支更关键

你担心诊断顺序是否固定，例如先看氧气纯度还是先看 V3 阀。老师的原意是：

```text
顺序没有那么关键。
谁排第一、谁排第二、谁排第三没有太大关系。
只要确实要走几个分支就可以。
```

可确认结论：

```text
第一版不必纠结深度优先还是宽度优先，先保证主要原因分支完整。
```

### 2.6 故障树不要形成环

你提到耦合问题。老师指出：

```text
树是往下分的，没有再交叉的部分。
如果形成环，就说明梳理得不对。
```

可确认结论：

```text
故障树表达上应保持无环结构；同一测点可以支撑多个判断，但原因路径不要互相循环。
```

## 6. 物料平衡在故障树中的位置

### 6.1 会议原文确认

老师明确说，物料平衡可以算一算，用来确认是否不平衡，但不能直接定位唯一原因。平衡方程更多判断“平不平衡”，不能直接判断氧气抽多还是抽少，还要结合质量指标，如氧气纯度。

### 6.2 执行扩展

建议把物料平衡作为“辅助校核节点”：

```text
疑似氮塞
→ 主塔/空气系统分支
→ 物料平衡校核
→ 判断空气量是否偏高/偏低
→ 再结合纯度、压力、阻力、阀位判断
```
